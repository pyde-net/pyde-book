# Roadmap

Pyde's path from design-complete to mainnet, structured as **five phases (MC-0 through MC-5)** with **three parallel implementation streams** in the core phase (MC-1). Each phase ships when its bar is met ŌĆö no calendar dates.

Coordination details (crate ownership, branching protocol, interface contracts, session handoff prompts for the three streams) live in [`companion/IMPLEMENTATION_PLAN.md`](companion/IMPLEMENTATION_PLAN.md). Read that first if you're implementing.

**Legend:**
- `[SEQ]` ŌĆö sequential (must complete before the next phase starts)
- `[PAR]` ŌĆö parallel (can run concurrently with siblings)
- `ŌåÆ` ŌĆö explicit dependency
- `╬▒` / `╬▓` / `╬│` ŌĆö owning implementation stream (see `IMPLEMENTATION_PLAN.md` ┬¦4)

---

## Top-level shape

```
MC-0  INTERFACE FOUNDATION              [SEQ ŌĆö main session]
  Ōöé   Create engine repo, lock types + interfaces crates, CI baseline.
  Ōöé   This is the prerequisite that makes parallelism safe.
  Ō¢╝
MC-1  PROTOCOL CORE                     [PAR ŌĆö three streams]
  Ōöé   Stream ╬▒ (Toolchain)   in pyde-net/otigen
  Ōöé   Stream ╬▓ (Execution)   in pyde-net/engine on `execution-side` branch
  Ōöé   Stream ╬│ (Consensus)   in pyde-net/engine on `consensus-side` branch
  Ō¢╝
MC-2  INTEGRATION                       [SEQ ŌĆö ╬│ owns]
  Ōöé   Merge ╬▓ + ╬│ branches; bring up local devnet end-to-end.
  Ō¢╝
MC-3  STATE SYNC + PARACHAIN ACTIVATION [SEQ ŌĆö ╬▓ + ╬│ joint]
  Ōöé   Snapshot machinery, weak-subjectivity, parachain framework live.
  Ō¢╝
MC-4  PERFORMANCE + FAILURE HANDLING    [PAR within]
  Ōöé   Performance harness, chaos drills, soak.
  Ō¢╝
MC-5  VALIDATION + MAINNET LAUNCH       [SEQ]
      External audits, incentivized testnet, mainnet.
```

Old MC-1 through MC-7 numbering (pre-2026-05-23) collapses into this shape: old MC-1 + MC-2 ŌåÆ MC-0 + MC-1; old MC-3 ŌåÆ folded into Stream ╬▒; old MC-4 ŌåÆ folded into MC-3; old MC-5 ŌåÆ MC-3; old MC-6 ŌåÆ MC-4; old MC-7 ŌåÆ MC-5.

#### Open questions to settle before MC-2 lands

1. ~~**`receipts_cf` + `txs_cf` ownership**~~ ŌĆö **resolved**: landed in state crate via PR [#132](https://github.com/pyde-net/engine/pull/132). Rationale: state already owns every persistent CF; node-owned CFs would mean either (a) two RocksDB instances or (b) a shared registry ŌĆö both add coordination burden.
2. **`consensus_store` schema** ŌĆö ╬│ has a `consensus_store` trait; what's the persistent layout? Vertex+round indexes, or vertices in RocksDB + indexes in memory?
3. ~~**WAL on the mempool**~~ ŌĆö **resolved: no WAL in v1.** Rationale (capital-allocation lens): the only crash window where a WAL would save a tx is "submitted via RPC + crash before gossip-broadcast." Gossip already provides redundancy ŌĆö if even one peer received the tx pre-crash, it survives (and wallets typically retry on no-receipt anyway). Adding a per-tx WAL trades a small UX edge case (re-submit-after-crash for a small set of solo-validator-RPC users) against write amplification on every admission and bootstrap complexity. v2 reconsiders if operator telemetry shows the crash-window cost is real. The MempoolStats counters (`admitted_local`, `admitted_gossiped`) are the canary ŌĆö if `admitted_local` grossly outpaces `admitted_gossiped` after MC-2 ships, the RPC-direct submission pattern is heavy and the case for WAL strengthens.
4. ~~**Base fee update cadence**~~ ŌĆö **resolved: per-wave update at commit time, EIP-1559 elasticity** (gas target = 50% of `BLOCK_GAS_MAX`). Rationale: Pyde's wave maps to Ethereum's "block" for fee-economics purposes ŌĆö each commit shifts base fee up/down by Ōēż12.5% based on `wave_gas_used / wave_gas_target` (the canonical EIP-1559 formula ╬▓.3 PR-2 already encodes). The node hooks into the consensus commit path: after `commit_with_events` returns, compute the new base_fee + call `WasmExecutorAdapter::update_base_fee(new)` + `PydeMempoolView::update_base_fee(new)` (both now interior-mutable per PR [#133](https://github.com/pyde-net/engine/pull/133) + PR [#129](https://github.com/pyde-net/engine/pull/129)). Two consumers updated in lockstep keep `Receipt::fee_paid` math and mempool `InsufficientBalance` admission threshold consistent. **Gas target choice (50%)** matches Ethereum's tuning since the EIP-1559 paper; v2 may tune up/down if Pyde's elastic-round timing makes 50% mis-targeted (e.g., if waves see consistent under- or over-utilisation).
5. **Light client cutoff** ŌĆö light client lands at MC-3 (state sync). For MC-2, `pyde light` could be a stub that returns "MC-3" rather than building partial.
6. **Two-pass `WaveCommitter.tick_for_testing` reinstatement** ŌĆö ╬│.15 / ╬│.16's 2 `#[ignore]`'d validator-integration tests need a multi-validator coordination harness; MC-2's 4+7-validator devnets are the natural test rigs.

## End-to-end flow: user ŌåÆ execution ŌåÆ user

For context on what all this protocol work enables, here's the full E2E flow once all chunks are landed:

```text
1. USER: opens wallet, builds tx (function call, args, gas budget)
2. WALLET: runs local wasmtime preview ŌåÆ shows state changes, gas estimate, events
3. USER: reviews preview, signs (FALCON-512)
4. WALLET: optionally encrypts under committee threshold key (Kyber-768)
5. WALLET ŌåÆ RPC: pyde_sendRawTransaction(signed_tx)
6. RPC: validates ingress (sig, balance, nonce, gas, chain_id)
7. RPC ŌåÆ MEMPOOL WORKER: forwards via libp2p
8. WORKER: adds to pending batch
9. WORKER: seals batch, gossipps to other workers, collects Ōēź85 certifications
10. WORKER ŌåÆ PRIMARY: certified batch_hash available for inclusion
11. PRIMARY: produces vertex with batch_hash in batch_refs (+ decryption shares if applicable)
12. VERTEX: gossipped via libp2p/gossipsub on pyde/vertices/1 topic
13. DAG: grows; each round adds 128 vertices
14. ANCHOR: deterministically selected via Hash(beacon, round, prev_root) mod 128
15. SUPPORT: round R+2's 85+ vertices transitively reference anchor ŌåÆ 3-stage support
16. COMMIT: subdag walk (BFS-for-set + canonical sort)
17. DECRYPT: batch threshold-decrypt all encrypted txs in subdag (shares already piggybacked)
18. SCHEDULE: hybrid scheduler (static access + Block-STM) partitions for parallel execution
19. EXECUTE: wasmtime runs each tx (per-tx overlays for isolation; success ŌåÆ merge, trap ŌåÆ discard)
20. STATE: changes accumulate in DashMap ŌåÆ JMT update ŌåÆ new state_root
21. SIGN: committee FALCON-signs (wave_id, blake3_root, poseidon2_root)
22. PERSIST: WaveCommitRecord synchronously to disk; vertices/batches/receipts lazily
23. FINALITY: 85+ sigs collected ŌåÆ HardFinalityCert formed
24. USER ŌåÉ RPC: pyde_getTransactionReceipt(tx_hash) returns success/revert + state changes + gas used
25. USER: sees confirmation in wallet UI

Total wall-clock from step 5 (submit) to step 25 (confirmation visible): ~500ms-1s under normal conditions.
```

Each step maps to specific chunks in the roadmap. The full path traverses MC-2 (consensus, execution, state, crypto, network, accounts, slashing) end-to-end, with MC-3 (otigen, SDKs, wallet) at the boundaries.

## Operating principle

The bias of this roadmap is **honesty over optimism**. No chunk ships before its bar is met. No item is checked off until the work behind it is actually done. If something turns out to be wrong, it gets honestly rewritten ŌĆö including this roadmap.

The work is the work. It ships when it is ready.
