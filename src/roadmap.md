# Roadmap

Pyde's path from design-complete to mainnet, structured as **five phases (MC-0 through MC-5)** with **three parallel implementation streams** in the core phase (MC-1). Each phase ships when its bar is met ÔÇö no calendar dates.

Coordination details (crate ownership, branching protocol, interface contracts, session handoff prompts for the three streams) live in [`companion/IMPLEMENTATION_PLAN.md`](companion/IMPLEMENTATION_PLAN.md). Read that first if you're implementing.

**Legend:**
- `[SEQ]` ÔÇö sequential (must complete before the next phase starts)
- `[PAR]` ÔÇö parallel (can run concurrently with siblings)
- `Ôćĺ` ÔÇö explicit dependency
- `╬▒` / `╬▓` / `╬│` ÔÇö owning implementation stream (see `IMPLEMENTATION_PLAN.md` ┬ž4)

---

## Top-level shape

```
MC-0  INTERFACE FOUNDATION              [SEQ ÔÇö main session]
  Ôöé   Create engine repo, lock types + interfaces crates, CI baseline.
  Ôöé   This is the prerequisite that makes parallelism safe.
  Ôľ╝
MC-1  PROTOCOL CORE                     [PAR ÔÇö three streams]
  Ôöé   Stream ╬▒ (Toolchain)   in pyde-net/otigen
  Ôöé   Stream ╬▓ (Execution)   in pyde-net/engine on `execution-side` branch
  Ôöé   Stream ╬│ (Consensus)   in pyde-net/engine on `consensus-side` branch
  Ôľ╝
MC-2  INTEGRATION                       [SEQ ÔÇö ╬│ owns]
  Ôöé   Merge ╬▓ + ╬│ branches; bring up local devnet end-to-end.
  Ôľ╝
MC-3  STATE SYNC + PARACHAIN ACTIVATION [SEQ ÔÇö ╬▓ + ╬│ joint]
  Ôöé   Snapshot machinery, weak-subjectivity, parachain framework live.
  Ôľ╝
MC-4  PERFORMANCE + FAILURE HANDLING    [PAR within]
  Ôöé   Performance harness, chaos drills, soak.
  Ôľ╝
MC-5  VALIDATION + MAINNET LAUNCH       [SEQ]
      External audits, incentivized testnet, mainnet.
```

Old MC-1 through MC-7 numbering (pre-2026-05-23) collapses into this shape: old MC-1 + MC-2 Ôćĺ MC-0 + MC-1; old MC-3 Ôćĺ folded into Stream ╬▒; old MC-4 Ôćĺ folded into MC-3; old MC-5 Ôćĺ MC-3; old MC-6 Ôćĺ MC-4; old MC-7 Ôćĺ MC-5.

#### Open questions to settle before MC-2 lands

1. ~~**`receipts_cf` + `txs_cf` ownership**~~ ÔÇö **resolved**: landed in state crate via PR [#132](https://github.com/pyde-net/engine/pull/132). Rationale: state already owns every persistent CF; node-owned CFs would mean either (a) two RocksDB instances or (b) a shared registry ÔÇö both add coordination burden.
2. **`consensus_store` schema** ÔÇö ╬│ has a `consensus_store` trait; what's the persistent layout? Vertex+round indexes, or vertices in RocksDB + indexes in memory?
3. ~~**WAL on the mempool**~~ ÔÇö **resolved: no WAL in v1.** Rationale (capital-allocation lens): the only crash window where a WAL would save a tx is "submitted via RPC + crash before gossip-broadcast." Gossip already provides redundancy ÔÇö if even one peer received the tx pre-crash, it survives (and wallets typically retry on no-receipt anyway). Adding a per-tx WAL trades a small UX edge case (re-submit-after-crash for a small set of solo-validator-RPC users) against write amplification on every admission and bootstrap complexity. v2 reconsiders if operator telemetry shows the crash-window cost is real. The MempoolStats counters (`admitted_local`, `admitted_gossiped`) are the canary ÔÇö if `admitted_local` grossly outpaces `admitted_gossiped` after MC-2 ships, the RPC-direct submission pattern is heavy and the case for WAL strengthens.
4. ~~**Base fee update cadence**~~ ÔÇö **resolved: per-wave update at commit time, EIP-1559 elasticity** (gas target = 50% of `BLOCK_GAS_MAX`). Rationale: Pyde's wave maps to Ethereum's "block" for fee-economics purposes ÔÇö each commit shifts base fee up/down by ÔëĄ12.5% based on `wave_gas_used / wave_gas_target` (the canonical EIP-1559 formula ╬▓.3 PR-2 already encodes). The node hooks into the consensus commit path: after `commit_with_events` returns, compute the new base_fee + call `WasmExecutorAdapter::update_base_fee(new)` + `PydeMempoolView::update_base_fee(new)` (both now interior-mutable per PR [#133](https://github.com/pyde-net/engine/pull/133) + PR [#129](https://github.com/pyde-net/engine/pull/129)). Two consumers updated in lockstep keep `Receipt::fee_paid` math and mempool `InsufficientBalance` admission threshold consistent. **Gas target choice (50%)** matches Ethereum's tuning since the EIP-1559 paper; v2 may tune up/down if Pyde's elastic-round timing makes 50% mis-targeted (e.g., if waves see consistent under- or over-utilisation).
5. **Light client cutoff** ÔÇö light client lands at MC-3 (state sync). For MC-2, `pyde light` could be a stub that returns "MC-3" rather than building partial.
6. **Two-pass `WaveCommitter.tick_for_testing` reinstatement** ÔÇö ╬│.15 / ╬│.16's 2 `#[ignore]`'d validator-integration tests need a multi-validator coordination harness; MC-2's 4+7-validator devnets are the natural test rigs.

### V1 reservations that create room for v2 features

V1 ships *interfaces*; v2 ships *implementations*. Discipline: don't reach into v2 while v1 is shipping, but reserve the protocol surfaces v2 needs so contracts written today survive the upgrade unchanged.

| v2 feature | v1 reservation | Cost at v1 |
| --- | --- | --- |
| Programmable accounts | `AuthKeys::Programmable` enum tag `0x03` | Enum variant, unused ÔÇö ~zero |
| Programmable accounts | Account `code_hash` + `storage_root` (unified with contracts) | Already shipped (account/contract account shape unified) |

| Parachains (further depth) | `cross_call` host fn, `HardFinalityCert` primitive, async callback slots | Already shipped (Chapter 13, `companion/PARACHAIN_DESIGN.md`) |

The discipline: every entry above is something the v1 protocol can ship for ~zero marginal cost, but skipping any one of them would force a hard-fork rewrite when v2 lands. Reserving them now is cheap insurance.

## End-to-end flow: user Ôćĺ execution Ôćĺ user

For context on what all this protocol work enables, here's the full E2E flow once all chunks are landed:

```text
1. USER: opens wallet, builds tx (function call, args, gas budget)
2. WALLET: runs local wasmtime preview Ôćĺ shows state changes, gas estimate, events
3. USER: reviews preview, signs (FALCON-512)
4. WALLET: optionally encrypts under committee threshold key (Kyber-768)
5. WALLET Ôćĺ RPC: pyde_sendRawTransaction(signed_tx)
6. RPC: validates ingress (sig, balance, nonce, gas, chain_id)
7. RPC Ôćĺ MEMPOOL WORKER: forwards via libp2p
8. WORKER: adds to pending batch
9. WORKER: seals batch, gossipps to other workers, collects Ôëą85 certifications
10. WORKER Ôćĺ PRIMARY: certified batch_hash available for inclusion
11. PRIMARY: produces vertex with batch_hash in batch_refs (+ decryption shares if applicable)
12. VERTEX: gossipped via libp2p/gossipsub on pyde/vertices/1 topic
13. DAG: grows; each round adds 128 vertices
14. ANCHOR: deterministically selected via Hash(beacon, round, prev_root) mod 128
15. SUPPORT: round R+2's 85+ vertices transitively reference anchor Ôćĺ 3-stage support
16. COMMIT: subdag walk (BFS-for-set + canonical sort)
17. DECRYPT: batch threshold-decrypt all encrypted txs in subdag (shares already piggybacked)
18. SCHEDULE: hybrid scheduler (static access + Block-STM) partitions for parallel execution
19. EXECUTE: wasmtime runs each tx (per-tx overlays for isolation; success Ôćĺ merge, trap Ôćĺ discard)
20. STATE: changes accumulate in DashMap Ôćĺ JMT update Ôćĺ new state_root
21. SIGN: committee FALCON-signs (wave_id, blake3_root, poseidon2_root)
22. PERSIST: WaveCommitRecord synchronously to disk; vertices/batches/receipts lazily
23. FINALITY: 85+ sigs collected Ôćĺ HardFinalityCert formed
24. USER ÔćÉ RPC: pyde_getTransactionReceipt(tx_hash) returns success/revert + state changes + gas used
25. USER: sees confirmation in wallet UI

Total wall-clock from step 5 (submit) to step 25 (confirmation visible): ~500ms-1s under normal conditions.
```

Each step maps to specific chunks in the roadmap. The full path traverses MC-2 (consensus, execution, state, crypto, network, accounts, slashing) end-to-end, with MC-3 (otigen, SDKs, wallet) at the boundaries.
